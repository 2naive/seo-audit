# Changelog — SEO Audit Skill

## [1.14.1] — 2026-04-09 — Обязательная самопроверка effort estimate (Шаг 13.7а)

Патч-релиз: на отчёте maxilac.ru v1.14.0 обнаружено, что агент сообщил `effortEstimate.stages[].taskCount = [9, 7, 1]`, тогда как реальное распределение `recommendations[]` по `phase` было `[8, 8, 1]`. Аналогично разошлись `totals` (139 vs пересчитанные 140). Renderer с v1.13.0 уже игнорирует данные агента и сам пересчитывает stages/totals из per-rec, поэтому **клиент видит правильные числа**, но сам JSON содержит неконсистентные данные.

Это паллиатив. Правильное решение — вынести Rule 13 в детерминированный модуль `estimate.js` (Вариант D), но это MAJOR-релиз с breaking change в JSON-схеме. Откладывается; полная аргументация и план реализации сохранены в разделе «Открытые архитектурные решения» в `CLAUDE.md`.

### Added — Шаг 13.7а «Обязательная самопроверка перед сохранением»

Новый подшаг в Правиле 13 SKILL.md. Перед записью JSON агент обязан прогнать четыре арифметических чека и **переписать** `effortEstimate.stages/totals` при любом несовпадении:

1. **Чек 1 — task counts**: для каждого `stage.id ∈ {1,2,3}` выполнить `stage.taskCount === recommendations.filter(r => phaseToStage(r.phase) === stage.id).length`. Где `phaseToStage = {urgent:1, month:2, strategy:3}`.
2. **Чек 2 — суммы часов по ролям**: для каждого этапа и каждой роли (`seo`, `dev`, `qa`, `devops`, `design`) сумма по входящим в этап рекомендациям должна совпадать с `stage.hours[role]`.
3. **Чек 3 — PM по этапу**: `stage.hours.pm === ceil(0.23 × (sum базовых ролей))` и `stage.hours.total === base + pm`.
4. **Чек 4 — totals**: `effortEstimate.totals[role] === sum(stages[].hours[role])` для всех 7 значений (5 ролей + pm + total).

Источник истины — `recommendations[].estimateHours` и `recommendations[].phase`. При конфликте сводки с per-rec данными — доверять per-rec, переписывать сводку.

В шаге явно сказано: **«Не "прикидывай в уме". Сделай это формально: выпиши массив значений по каждой роли, сложи, сравни.»** — потому что LLM-арифметика без выписывания промежуточных сумм даёт ошибку.

### Added — раздел «Открытые архитектурные решения» в CLAUDE.md

Сохранён полный контекст обсуждения Варианта D (вынос расчётов в `estimate.js`) для будущих итераций: проблематика, целевая архитектура, что выигрывается, открытые подвопросы реализации, причина откладывания. Чтобы при возврате к этой задаче не пришлось заново обсуждать.

### Files

- `SKILL.md` — добавлен Шаг 13.7а с четырьмя чеками. `skillVersion: 1.14.1`.
- `CLAUDE.md` — версия `1.14.1`. Новый раздел «Открытые архитектурные решения» с планом Варианта D на ~50 строк.
- `generate-report.js` — `SKILL_VERSION = '1.14.1'`. Логика рендерера не меняется (он уже самостоятельно пересчитывает stages/totals с v1.13.0).

### Не входит в этот патч (отложено)

- Фикс false-positive в AEO-секции: `aeoReadiness.firstParagraphWords: 1` для всех страниц трактуется как «короткий лид-абзац» (зелёное), хотя 1 слово — это отсутствие лида. Логичный фикс: считать «коротким» при `>= 5 && <= 60`, либо ужесточить collector.
- Грамматика «1 слов» в плитке AEO — нужно склонение через helper.

Эти два дефекта тоже были обнаружены аудитом v1.14.0, но не входят в скоуп v1.14.1 (вариант B по обсуждению с оператором).

## [1.14.0] — 2026-04-09 — Реальная секция AEO / GEO в отчёте

Минорный релиз: методология заявляла «AEO/GEO готовность для AI-поиска» как часть проверки, но в самом отчёте никакой видимой секции под это не было. Данные собирались (`siteData.llmsTxt`, `siteData.aiCrawlers`, `pages[].metrics.aeoReadiness`), но никогда не отображались — клиент не мог их увидеть. Это нарушало принцип «что заявлено — то показано», поднятый в Правиле 15 (целостность данных).

### Added — секция «AEO / GEO — готовность к AI-поиску»

Новая карточка в отчёте между «Анализ типов страниц» и «Технические проверки». Источник данных — то, что уже собирается collector-ом в Фазе 1, без необходимости менять JSON-схему. В TOC добавлен соответствующий пункт.

Содержит четыре блока:

1. **Краткие плитки-метрики** (4 шт.):
   - `llms.txt` — найден / отсутствует, упоминание `llms-full.txt`.
   - AI-краулеры — `N заблокировано` / `N разрешено` / `не настроены`, плюс счётчик «не упомянуты».
   - FAQ-секции — `N / total` страниц с FAQ-блоком.
   - Первый абзац — средняя длина в словах, цвет по норме (≤ 60 — зелёный, > 60 — оранжевый).

2. **Статус AI-краулеров** (если `siteData.aiCrawlers` непустой) — три ряда тегов:
   - 🔴 «Заблокированы:» список (красный)
   - 🟢 «Явно разрешены:» список (зелёный)
   - 🟡 «Не упомянуты в robots.txt:» список (оранжевый) с подсказкой
   - Под таблицей — короткая enlightenment-строка про значение «не упомянуты».

3. **Готовность контента страниц** (если в `pages[]` есть хотя бы одна страница с `aeoReadiness`):
   - Страниц с первым абзацем ≤ 60 слов: `N / total`
   - Страниц с длинным первым абзацем (> 60 слов): `N / total`
   - Страниц с FAQ-секцией: `N / total`
   - Каждая строка с цветовой подсветкой и колонкой «Что это значит для AI-поиска».

4. **Заключительный параграф** — короткое объяснение, какие сигналы AEO/GEO формируют готовность к AI-обзорам (открытый доступ AI-краулерам, llms.txt, краткие лид-абзацы, FAQ + Schema.org).

### Behavior

- Секция **не отображается**, если ни одного AEO-сигнала нет (нет `siteData`, нет страниц с `aeoReadiness`). Это обеспечивает graceful degradation для старых отчётов, где collector ещё не заполнял эти поля.
- Подсчёт страниц с длинным/коротким лидом и FAQ ведётся **только** по тем страницам, у которых `metrics.aeoReadiness` присутствует (защита от ложного 0/N в случае пропуска collector-а).

## [1.13.2] — 2026-04-09 — Целостность данных в карточках страниц

Патч-релиз: критический баг рендера + новое Правило 15 о согласованности `metrics` ↔ `issues[]`. Обнаружен на отчёте maxilac.ru: для главной страницы метрика H1 показывалась зелёной галочкой (✓), хотя ниже в issues стояло «H1 отсутствует полностью (контент H1 реализован через H2)». Причина — в JS пустой массив `[]` truthy, и в `generate-report.js` строка `${metricRow('H1', m.h1 ? '✓' : 'нет', !!m.h1)}` всегда давала зелёный для существующего поля.

### Fixed — H1-метрика читает массив корректно

- В `generate-report.js` (`pageCard`) переписана нормализация `m.h1`: всегда массив, проверка по `length`, отображается реальный текст H1 (truncate до 60 символов) вместо безликой галочки. Если H1 нет — красное «нет». Если несколько — красное «N шт. — дубль».
- Заодно улучшены метрики Title/Description: вместо просто «77 симв.» показывается «77 симв. (длинно)» при выходе из диапазона. Это убирает соблазн поверить нейтральному числу.

### Added — Правило 15 «Целостность данных в pages[]»

Новое правило в SKILL.md, фиксирующее три уровня консистентности:

1. **Источник истины — collector**. Все поля `pages[].metrics` берутся только из результата JS-collector в Фазе 1. Никаких ручных правок «по памяти».
2. **Контракт по типам полей**. Жёсткая таблица: `h1` — массив строк (всегда `[]`, никогда не строка/null); `canonical` — `string|null`; `schemaTypes` — `string[]`; счётчики — числа; флаги — boolean. Объяснено почему `h1` обязан быть массивом (truthy/falsy ловушка JS).
3. **Согласованность `metrics` ↔ `issues[]`**. Таблица соответствия: каждый critical/warning issue должен подтверждаться конкретным состоянием поля metrics. Если в issues «H1 отсутствует» → `metrics.h1 === []`. Если в issues «N изображений без alt» → `metrics.imgsNoAlt === N`. При конфликте — доверять collector-у, переписывать issue.
4. **Согласованность `pages[]` ↔ `recommendations[]`**. Любая рекомендация уровня страницы должна ссылаться через `affectedUrls[]` или `sourceChecks[]` на страницу с подтверждающим состоянием metrics. Не выдумывать рекомендации без подтверждения.
5. **Самопроверка перед сохранением JSON** — короткий чек-лист (тип полей, подтверждение каждого issue, отсутствие противоречий, разумность общей картины).

### Fixed — JSON-схема pages[].metrics.h1

В примере JSON-схемы поле `h1` было задокументировано как строка (`"h1": "..."`), хотя collector возвращает массив. Исправлено на `["Заголовок страницы"]` — соответствует реальности и Правилу 15.

### Files

- `generate-report.js` — `pageCard` переписан в части метрик H1/Title/Description, новая нормализация `h1Arr`/`h1Display`/`h1Ok`. `SKILL_VERSION = '1.13.2'`.
- `SKILL.md` — добавлено Правило 15 (~50 строк, 4 таблицы), исправлен пример `h1` в схеме pages[]. `skillVersion: 1.13.2`.
- `CLAUDE.md` — версия 1.13.2.

## [1.13.0] — 2026-04-09 — Перестановка секций отчёта и детализация задач по этапам

Минорный релиз: блок оценки трудозатрат переехал ниже Плана действий, «Топ-10 по RICE» заменён на детальный список задач каждого этапа с часами на каждую задачу. Сводка теперь гарантированно консистентна с Планом действий.

### Changed — порядок секций в отчёте

- **План действий теперь идёт перед оценкой трудозатрат**, не после. Раньше клиент сначала видел сводную таблицу часов и Топ-10, и только потом сам план — это путало приоритет восприятия. Теперь план — основной артефакт, оценка — приложение к нему.

### Changed — Сводка по этапам строго совпадает с Планом действий

- Группировка задач в этапы (`stage 1/2/3`) теперь использует **только** поле `phase` из Roadmap (`urgent`/`month`/`strategy`), без дополнительных условий по `severity`/`priority`. Это гарантирует: количество задач в каждом этапе сводки = количество задач в одноимённой фазе Плана действий, а суммарные часы в `totals` покрывают **все** рекомендации отчёта.
- Сам генератор отчёта (`generate-report.js`) теперь сам считает часы по этапам из `recommendations[].estimateHours`, а не верит на слово полю `effortEstimate.stages[].hours` от агента — это устраняет рассинхрон между разделами.
- PM в каждом этапе пересчитывается как `ceil(0.23 × сумма базовых ролей)` из тех же данных.

### Changed — «Топ-10 по приоритету» заменён на «Задачи по этапам с часами»

- Удалён `effortEstimate.topByRice[]` из JSON-схемы и Шага 13.8 SKILL.md.
- Вместо одной таблицы из 10 строк отчёт теперь показывает **все задачи**, сгруппированные по 3 этапам, с конкретным числом часов на каждую задачу. Каждая строка кликабельна и ведёт на детализацию рекомендации (`#rec-N`).
- Это даёт клиенту полную картину: сколько часов занимает каждая отдельная правка в каждом этапе, без необходимости пересчёта или ручного сопоставления Топ-10 с детализацией.

### Removed — дисклеймер «оценка ориентировочная»

- Удалён жёлтый блок дисклеймера в конце секции трудозатрат: «оценка ориентировочная», «реальные часы зависят от...», «стоимость согласований». Эти формулировки прямо снижают доверие клиента и звучат как страховка на случай провала.
- Из SKILL.md удалена соответствующая инструкция «В отчёте указывай: оценка ориентировочная...». Числа подаются как итог экспертной оценки.

### Changed — TOC

- Ссылка «Оценка трудозатрат» в оглавлении больше не содержит число часов (раньше — устаревшее поле `withReserve`/`subtotal`). Теперь просто заголовок.

### Files

- `generate-report.js` — переписана функция `effortEstimateHtml`: своя группировка задач по этапам, детальный список вместо Top-10, удалён дисклеймер. Поменян порядок `${roadmapHtml}` ↔ `${effortEstimateHtml}`.
- `SKILL.md` — Шаг 13.6: группировка строго по `phase`. Шаг 13.8: удалён `topByRice`, добавлен запрет на формулировку «оценка ориентировочная». JSON-пример очищен от `topByRice`.

## [1.12.0] — 2026-04-09 — Переработка блока оценки трудозатрат

Минорный релиз: смета теперь подаётся как профессиональный документ от подрядчика, а не как автоматический расчёт.

### Changed — целые часы вместо дробных и диапазонов

- Все часы округляются вверх (`Math.ceil`) до целого. Минимум — 1 час, если роль участвует. Дробные числа («1.5», «2.95») и нули вместо тире удалены.
- `effortHours` (текстовое поле в карточке рекомендации) — однозначное значение «`4 часа`» / «`12 часов`», без диапазонов «1–2 часа».
- Baseline-таблица обновлена под целые часы (минимум 1 для single, 2 для template).

### Changed — резерв встроен в часы каждой задачи

- Добавлен новый множитель `M_reserve` (1.15–1.40 по объёму сайта) применяется к baseline вместе с остальными множителями
- Из JSON-схемы удалены: `reservePercent`, `reserveHours`, `withReserve`
- Из HTML удалена строка «Резерв на неопределённость (X%) — landing проект» — клиент видит честные итоговые числа без намёков на неуверенность

### Changed — PM как полноценная роль

- Из JSON-схемы удалено: `managementHours`
- PM теперь колонка в каждом этапе таблицы
- Расчёт: `stage.pm = ceil(0.23 × (seo + dev + qa + devops + design))` — 23% от объёма работ остальных ролей

- Кириллический slug: `/новости/некий-заголовок` → `/новости/{slug}`
- Hash/UUID: `/page/a1b2c3d4` → `/page/{hash}`
- Остальное оставляется как есть (статичные пути)

Группировка по pattern, сортировка по `matchedCount` desc, лимит 20.

Если sitemap.xml недоступен — fallback на внутренние ссылки с главной.

## [1.8.1] — 2026-04-09

### Fixed — PDF generation работает при активном Claude Chrome Extension

**Корневая причина** (диагностика):
- На Windows когда работает Claude Chrome Extension, обычный `chrome.exe --headless --print-to-pdf` молча падает (exit 0, файл не создан, `lockfile: Device or resource busy`).
- Process Singleton mechanism Chrome: `chrome.exe` видит существующий main process и выводит «Opening in existing browser session» — подсасывает к нему вместо запуска изолированной headless instance.
- `--user-data-dir=$(mktemp -d)` НЕ помогает обойти Singleton (вопреки распространённому совету).
- Только `--remote-debugging-port=N` обходит Singleton — но `--print-to-pdf` несовместимо с debug port (флаги взаимоисключающие).

**Решение — CDP через WebSocket** (как в `chrome-launcher`/Lighthouse):
1. `spawn` Chrome с `--remote-debugging-port=N`, `detached: true`, `child.unref()` (наследование handles от родителя ломает Singleton bypass)
2. Polling `GET http://127.0.0.1:N/json/version` до готовности (до 15 секунд)
3. `GET /json/list` — получить `webSocketDebuggerUrl` для page target
4. WebSocket handshake (HTTP Upgrade с `Sec-WebSocket-Key`)
5. Отправка `Page.enable` через WebSocket frame (с маскированием от клиента)
6. Пауза 2.5 сек для рендера (картинки, шрифты, base64 ассеты)
7. `Page.printToPDF` через CDP — получаем base64 в response
8. Сохранение через `Buffer.from(data, 'base64')`
9. Cleanup: `child.kill()` + `rmSync` временного профиля

**Реализация**:
- Минимальный WebSocket клиент написан с нуля через `http` + `net` + `crypto` — без npm-зависимостей (~100 строк)
- Поддерживает фрагментированные frames (7/16/64-bit length)
- Маскирование payload от клиента к серверу (требование RFC 6455)
- Обработка только text frames (opcode 1) — достаточно для CDP

**Тест**: PDF 3.4 MB сгенерирован при активном Claude Chrome Extension в текущей сессии. Время: ~5 секунд (включая запуск Chrome, навигацию, рендер, печать).

### Changed
- `generate-report.js`: функция `generatePDF()` async, использует CDP вместо `--print-to-pdf` флага
- Удалено известное ограничение из v1.8.0 «PDF не работает при активном Extension» — теперь работает
