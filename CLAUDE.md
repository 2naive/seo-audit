# Правила работы с SEO Audit Skill

## Текущая версия: 1.15.0

Этот файл читается автоматически при любой работе с файлами в этой директории.  
**Соблюдай все правила ниже** при любых изменениях — даже незначительных.

---

## Версионирование

Версия хранится в **двух местах** — оба обязательно обновлять синхронно:
1. `generate-report.js` — константа `SKILL_VERSION = '...'` (строка ~11)
2. `SKILL.md` — поле `"skillVersion": "..."` в JSON-схеме Фазы 3

### Когда что менять

| Тип изменения | Версия |
|---------------|--------|
| Новая фаза, новый раздел отчёта, новый инструмент | MINOR (1.**4**.0) |
| Исправление бага, уточнение инструкции, новая проверка | PATCH (1.3.**3**) |
| Переработка архитектуры, смена схемы JSON | MAJOR (**2**.0.0) |

## Обязательный порядок при любом изменении

Выполни все 4 шага. Пропустить нельзя.

### Шаг 1 — Проверь текущее состояние
```bash
cd "$(git rev-parse --show-toplevel)/.claude/skills/seo-audit"
git log --oneline -5
```
Убедись, что работаешь на актуальном коде.

### Шаг 2 — Внеси изменения в файлы

### Шаг 3 — Обнови версию
Определи новую версию по таблице выше. Затем:

**В `generate-report.js`** (строка ~11):
```js
const SKILL_VERSION = 'X.Y.Z';   // ← новая версия
```

**В `SKILL.md`** (поле skillVersion в Фазе 3):
```json
"skillVersion": "X.Y.Z",
```

**В `CLAUDE.md`** (эта строка вверху):
```
## Текущая версия: X.Y.Z
```

### Шаг 4 — Обнови CHANGELOG.md и закоммить
Добавь запись в начало `CHANGELOG.md` по шаблону:

```markdown
## [X.Y.Z] — YYYY-MM-DD

- **Заголовок проблемы** — описание root cause и что именно изменилось
```

Затем закоммить:
```bash
git add -A
git commit -m "тип: краткое описание

- детали изменения 1
- детали изменения 2

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>"
```

## Принцип обратной совместимости

**Новые изменения не должны нарушать ранее работающий функционал и не должны удалять существующие проверки или поля без явного подтверждения оператора.**

Перед внесением любого изменения:

1. **Прочитай** `CHANGELOG.md` — убедись, что понимаешь что уже работает
2. **Проверь** что изменение не удаляет, не переименовывает и не меняет семантику существующих полей JSON-схемы, существующих секций отчёта, существующих фаз SKILL.md
3. **Если нужно удалить или переименовать** — явно сообщи оператору что именно будет удалено и почему, дождись подтверждения перед коммитом

### Что считается нарушением совместимости (требует подтверждения)
- Удаление поля из JSON-схемы (`report-data.json`)
- Переименование поля в JSON-схеме
- Удаление или переименование секции в HTML-отчёте
- Удаление фазы или шага из `SKILL.md`
- Изменение формата имён выходных файлов
- Удаление технической проверки из списка `technical`

### Что НЕ требует подтверждения
- Добавление новых полей в JSON-схему (обратно совместимо)
- Добавление новых секций в отчёт
- Добавление новых фаз и шагов в `SKILL.md`
- Уточнение инструкций без изменения структуры
- Исправление ошибок в коде без изменения внешнего поведения

## Что НЕ делать

- Не изменять `SKILL_VERSION` в `generate-report.js` без записи в `CHANGELOG.md`
- Не коммитить без обновления версии (если были изменения в логике)
- Не использовать `git add -A` без проверки что в `seo-audit-output/` нет лишних файлов
- Не оставлять `scoreDetails` пустыми массивами в JSON отчёта
- Не использовать относительные пути `seo-audit-output/` — только абсолютные через `${OUTPUT_DIR}/`
- Не удалять и не переименовывать существующие поля/секции без подтверждения оператора

## Открытые архитектурные решения

Раздел для отложенных решений по архитектуре, к которым стоит вернуться. Не правила — а контекст для будущих итераций.

### Перенос Rule 13 (effort estimate) из SKILL.md в детерминированный модуль `estimate.js`

**Статус**: отложено. Решено в обсуждении v1.14.1, реализация позже.

**Проблема**. Rule 13 заставляет LLM-агента выполнять около десяти шагов чистой арифметики: lookup baseline-таблицы по `taskType × scope`, применение пяти множителей (`M_scale`, `M_cms`, `M_risk`, `M_qa`, `M_reserve`), распределение по 6 ролям через таблицу процентов, расчёт RICE, группировка в этапы по `phase`, расчёт PM как `ceil(0.23 × base)`, финальные суммы. LLM регулярно ошибается в этих шагах — самый показательный пример это v1.14.0 на отчёте maxilac.ru, где агент сообщил `stages.taskCount = [9, 7, 1]`, а реальное распределение по `phase` было `[8, 8, 1]`.

Параллельно `generate-report.js` уже **с v1.13.0** сам пересчитывает stages и totals из `recommendations[].estimateHours` и игнорирует то, что в JSON писал агент. То есть в системе фактически два калькулятора: один (агент) ошибается, второй (renderer) считает правильно, а клиент видит правильные числа только потому, что rendered выигрывает.

**Что делает агент vs что может делать код**:

| Решение | Источник |
|---|---|
| `taskType` (16 значений) | LLM — судит «эта рекомендация про meta_tags / canonical / schema?» |
| `scope` (single / template / site_wide) | LLM — судит «правка одной страницы или шаблона?» |
| `confidence` (0.5 / 0.8 / 1.0) | LLM — судит «доказательство в sourceChecks или гипотеза?» |
| baseline lookup, все множители, role-распределение, RICE, group by phase, PM, totals | **Чистая арифметика** — должно быть кодом |

LLM делает 3 решения и 7 шагов арифметики. Все ошибки — в арифметике.

**Целевая архитектура (Вариант D)**:

```
Phase 1 (collector)        → siteData + pages[].metrics    [код, уже так]
Phase 2 (анализ)           → technical[] + scoreDetails    [LLM, уже так]
Phase 3 (рекомендации)     → recommendations[] с полями:   [LLM]
                             title, description, impact, fix,
                             steps, verify, priority, difficulty,
                             taskType, scope, confidence,
                             affectedUrls, sourceChecks
                             ↑ ТОЛЬКО аннотация, никакой математики
                             
Phase 3.5 (НОВОЕ, код)     → estimate.js обогащает каждый  [код]
                             rec полями estimateHours{},
                             riceScore, phase, stage.
                             stages[] и totals считаются
                             на лету в рендерере (как сейчас)
                             
Phase 4 (renderer)         → HTML/PDF                       [код]
```

**Что выигрывается**:
- SKILL.md худеет на ~150 строк (Шаги 13.2–13.7 + таблицы баз и множителей удаляются).
- Невозможен рассинхрон в принципе — нет двух калькуляторов.
- Тестируемость: для `estimate.js` можно писать unit-тесты `input rec → expected hours`.
- Калибровка под команду — правка одной const-таблицы, без переучивания LLM.
- Phase 3 быстрее и дешевле: агент пишет десятки полей меньше на каждую из 17 рекомендаций.
- `effortHours: "5 часов"` (текстовая строка) тоже становится derivable, рендерер склоняет «час/часа/часов» сам.

**Открытые подвопросы для реализации**:
1. `estimate.js` — отдельный модуль (рекомендация: да) или внутри `generate-report.js`?
2. Промежуточный JSON — писать обогащённую версию обратно в `report-data-*.json` (рекомендация: да, для отладки) или держать в памяти?
3. Удалять ли `recommendations[].effortHours` (текстовое поле) — рендерер склоняет сам? (рекомендация: да)
4. Baseline-таблицы — const в `estimate.js` (рекомендация: да) или отдельный `baseline.json`?
5. Это **MAJOR-релиз** (v2.0.0) — меняется JSON-схема (`effortEstimate.stages/totals` исчезают, `recommendations[].estimateHours` становится derived). Нужно явное подтверждение оператора по правилу обратной совместимости.

**Почему отложено**: в v1.14.1 выбран Вариант B — оставить `effortEstimate.stages/totals` в JSON и добавить агенту обязательную самопроверку (Шаг 13.7а в SKILL.md). Это сохраняет схему и не требует мажорного релиза. Но это паллиатив: самопроверки LLM ненадёжны (если бы они работали, баг 9/7/1 не появился бы), и Вариант D остаётся правильным архитектурным шагом, к которому надо вернуться при первом удобном случае — когда будет готовность к v2.0.0 и явное «да» оператора на breaking change в JSON-схеме.

**Связанные источники**: `Оценка/00-выводы-для-skill.md` — изначальное исследование оценки трудозатрат, где предлагался отдельный модуль `estimate.py` и `baseline.yaml`. Идея верная, реализация откладывается.
